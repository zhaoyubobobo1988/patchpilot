针对单次模型仿真耗时较长的问题，设计基于 task_id 的多任务管理机制：后端将任务提交、后台执行、状态查询、结果获取拆分为独立接口，并用线程锁保护共享任务状态；前端通过轮询展示任务进度，使用户在等待某个任务执行时仍可继续编辑其他场次并提交新的增强任务。


支持，而且你这个判断是对的：现在项目更像 **“MCP 工具 + Agent 运行环境”**，还不是完整意义上的 **领域智能体系统**。

我看了两个目录后，大致结论是：

`D:\code\RealTimeForecast` 已经像一个水文预报服务，包含 FastAPI、推理引擎、数据库写入、实时数据合并、MCP 可调用接口。

`D:\code\myopenclaw` 主要是 Hermes/OpenClaw 的容器化部署、MCP server 接入、备份、配置和网关。它可以让大模型调用 `run_forecast`、`get_latest_forecast`、`get_mswep_precip`、`get_timeseries_combined` 这些工具，但目前还没有形成你自己的“智能体大脑”。

真正的智能体，不只是“LLM + 工具”。它至少要解决这几个问题：

1. **目标是什么**
   
   智能体要有明确任务边界，比如：
   
   - 监测某个流域未来 48 小时洪水风险
   - 自动判断是否需要重新预报
   - 比较多模型结果差异
   - 发现数据异常并解释原因
   - 生成预警建议和日报
   - 在不确定时向人请求确认

2. **它知道什么**
   
   现在工具能查数据，但智能体缺少稳定的知识层。需要有：
   
   - 流域、测站、模型、阈值、业务规则的领域知识库
   - 历史预报记录、历史误差、典型洪水事件
   - 工具使用说明和约束
   - 用户偏好，比如报告格式、关注站点、预警口径

3. **它记得什么**
   
   你提到“记忆层”，这个确实缺。建议拆成三类：
   
   - **短期记忆**：当前对话、当前任务、当前调用过的工具结果
   - **长期记忆**：用户偏好、站点配置、历史判断、常用分析流程
   - **事件记忆**：每次预报、异常、预警、人工确认、后续结果验证

4. **它如何思考和行动**
   
   真正的 Agent 应该有一个循环：
   
   ```text
   接收目标
   → 理解当前状态
   → 制定计划
   → 选择工具
   → 执行工具
   → 观察结果
   → 判断是否继续
   → 输出结论或请求人工介入
   → 写入记忆
   ```

   你现在主要停在“LLM 可以调用工具”这一层，还没有自己的 planner、状态机、任务上下文和执行策略。

5. **它如何判断好坏**
   
   水文预报场景尤其需要评估层。比如：
   
   - 数据是否完整
   - MSWEP / GFS / IFS 哪个缺失
   - 预报峰值是否超过阈值
   - 多模型是否分歧很大
   - 与实测流量误差是否异常
   - 是否需要重跑或人工复核

6. **它如何保证安全**
   
   预报类智能体不能随便“自信输出”。需要：
   
   - 工具调用参数校验
   - 高风险结论加置信度
   - 预警发布前人工确认
   - 所有工具调用和推理过程留痕
   - 不允许模型编造数据来源

我建议你把项目升级成这样一套架构：

```text
用户 / 定时任务 / 外部事件
        ↓
Agent Orchestrator 智能体编排层
        ↓
Planner 任务规划
        ↓
Memory 记忆层
        ↓
Tool Registry 工具注册表
        ↓
MCP Tools / FastAPI / 数据库 / 预报模型
        ↓
Evaluator 结果评估
        ↓
Reporter 报告/预警生成
        ↓
Audit Log 过程留痕
```

具体要做的工作，我建议分 5 个阶段。

**第一阶段：定义领域智能体**
先不要急着堆技术。先定义它是什么 Agent。

比如可以叫：

`HydroForecastAgent`

核心职责：

- 每 3 小时自动检查最新数据
- 触发实时预报
- 比较 SimpleLSTM、Java、SMS3 等模型结果
- 判断洪峰、峰现时间、超警风险
- 检查输入数据缺口
- 生成结构化分析报告
- 必要时请求人工确认

**第二阶段：建设工具层**
你已有 MCP，但工具描述还可以继续增强。每个工具要声明：

- 什么时候使用
- 输入参数格式
- 返回字段含义
- 失败时如何处理
- 是否会写数据库
- 是否属于高成本/高风险操作

现在已有工具可以保留：

- `run_forecast`
- `get_latest_forecast`
- `get_mswep_precip`
- `get_timeseries_combined`

建议新增工具：

- `check_data_quality`
- `compare_model_outputs`
- `get_forecast_history`
- `compute_warning_level`
- `generate_forecast_report`
- `record_agent_decision`

**第三阶段：建设记忆层**
推荐先用 SQLite 或 Postgres，不要一开始就搞复杂向量库。

至少建这些表：

- `agent_sessions`：每次任务会话
- `agent_steps`：每一步计划、工具调用、观察结果
- `agent_memories`：长期记忆
- `forecast_cases`：一次预报案例
- `user_preferences`：用户偏好
- `domain_rules`：预警阈值、业务规则
- `decision_logs`：智能体做出的判断和理由

向量库可以后面加，用来检索历史事件、报告、文档。

**第四阶段：建设智能体循环**
这部分是最关键的。不要只依赖 Hermes/OpenClaw 的外壳，而是在你自己的项目里做一个明确的 Agent runtime。

建议核心模块：

```text
agent/
  orchestrator.py      # 主循环
  planner.py           # 任务规划
  memory.py            # 记忆读写
  tools.py             # 工具注册和调用
  evaluator.py         # 结果评估
  reporter.py          # 报告生成
  policies.py          # 安全策略
  schemas.py           # Pydantic 数据契约
```

一个最小循环可以是：

```text
目标：检查 21401550 站未来 48h 风险

1. 查询最近预报
2. 判断是否过期
3. 如过期，调用 run_forecast
4. 获取降雨和实测数据
5. 检查数据完整性
6. 计算洪峰、峰时、模型分歧
7. 生成风险等级
8. 写入记忆和审计日志
9. 输出报告
```

**第五阶段：做评估和闭环**
智能体不能只“说得像”。它要能被评估。

你需要设计测试集：

- 正常数据场景
- MSWEP 缺失场景
- GFS 缺失场景
- 多模型分歧场景
- 峰值超阈值场景
- 数据库不可用场景
- 工具调用失败场景

然后评估：

- 是否选对工具
- 是否发现异常
- 是否给出正确风险等级
- 是否引用真实数据
- 是否没有编造
- 是否正确请求人工确认

一句话总结：

你现在已经有了 **“可被智能体调用的预报工具”**，下一步要做的是 **“围绕这个工具建立一个有目标、有记忆、有规划、有评估、有安全边界的水文预报智能体”**。

最优先的三件事我建议是：

1. 在 `myopenclaw` 或新仓库里新增 `hydro_agent` 智能体核心模块。
2. 用 SQLite/Postgres 做第一版记忆和审计日志。
3. 实现一个固定工作流：`检查数据 → 运行预报 → 评估风险 → 生成报告 → 写入记忆`。

这样它就开始从“会调用工具的聊天模型”，变成“有业务闭环的领域智能体”。
